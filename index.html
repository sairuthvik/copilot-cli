Your first session

A 10-minute path from zero to useful

This is the onboarding flow I would recommend for any PM opening Copilot CLI for the first time.

1

Open the CLI

Launch Copilot in a project folder so it can orient itself around a real codebase and workflow.

# in your project folder
$ copilot
2

Authenticate once

Use your GitHub login so the CLI can work with the repos and workflows you already use.

# first session
> /login
✓ logged in
3

Ask a discovery question

Start with an area that matters to your roadmap so the value feels immediate and relevant.

# example
> How does our notifications feature work today?
4

Try planning mode

Use plan mode before asking for code. This is ideal for feature thinking, dependencies, and delivery conversations.

# switch modes
Shift+Tab
> Add CSV export for user reports
5

Save or share the outcome

Export the result so it can feed sprint planning, design reviews, or stakeholder updates.

# publish findings
> /share
What to type

The command stack a PM will actually use

Not every command matters equally. These are the ones most likely to create value in product planning, reviews, and cross-functional communication.

/plan PM core

Turn product asks into implementation plans

Use this when you want to discuss approach and trade-offs before anyone starts coding.

  • Best for feature scoping and engineering handoff prep
  • Use after pressing Shift+Tab to enter plan mode
  • Great prompt: “Add CSV export for account admins”
/review PM core

Understand what changed in a PR

Use it to identify user-facing impact, risk areas, and decision points before a release or demo.

  • Best for release reviews and demo prep
  • Follow with: “Explain that in plain English”
  • Great prompt: “What should I worry about in this PR?”
/research PM core

Research implementation patterns and options

Use GitHub and web sources to frame discovery work and compare approaches before roadmap commitment.

  • Best for exploratory product discovery
  • Helpful for build-vs-buy conversations
  • Great prompt: “How do SaaS products handle real-time collaboration?”
/share Everyone

Turn findings into something reusable

Export key conclusions to Markdown or a Gist so the rest of the team can consume them without replaying the session.

  • Best for sprint planning and stakeholder circulation
  • Pairs well with plans, research, and reviews
  • Great final step after a useful session
Real workflows

How PMs can use it day to day

Think of these as reusable plays. Click through them like scenario cards rather than reading one long wall of text.

Use case: Monday morning sprint planning

You want to know whether a feature is “close enough to real” to bring into the sprint and where delivery risk actually sits.

Sprint prep session
>What is the current state of notifications in this codebase?
Email notifications are implemented. Push notifications are scaffolded but not integrated with a provider.
>How much work is Slack notification support?
Likely small-to-medium scope: provider integration, settings UI, and permission checks.
>/plan Add Slack notification support
Plan created: dependencies, likely files, rollout notes, and risk areas included.

Use case: PR review before a demo

You do not need code-level mastery. You need to know what changed, what matters to users, and whether the story is safe to tell externally.

PR review session
>/review
Reviewing checkout changes, tests, dependency updates, and failure states...
Key improvements: faster checkout, cleaner request flow, fewer redundant API calls.
Potential issue: a failed payment can leave the user with no clear recovery path.
>Explain that issue in plain English
If payment fails, the user may see a broken experience instead of an actionable error message.

Use case: early discovery for a roadmap bet

You want to explore solution patterns and likely implementation shapes before taking a recommendation into planning.

Discovery research session
>/research How do modern SaaS products implement collaborative editing?
Researching GitHub repos, implementation patterns, and web sources...
Common patterns: CRDTs, operational transforms, WebSocket synchronization.
Common tooling: Yjs, ShareDB, Liveblocks.
>/share
Summary exported for team discussion.

Use case: explain technical choices to business stakeholders

This is where Copilot CLI becomes a translator: it can convert technical implementation into concise business impact.

Stakeholder prep session
>Explain our GraphQL migration in business terms
Today: the dashboard makes many network calls and loads slowly.
After migration: fewer round trips, faster dashboard loads, simpler feature delivery.
Impact: better user experience, improved retention potential, lower maintenance cost.
>/share
Executive summary exported.
Recommended ramp

Your first-week PM playbook

If you want a simple adoption plan, use this. It turns the CLI from “interesting tool” into a repeatable product habit.

Day 1

Learn the shape of the tool

Explore one repo, log in, and ask broad discovery questions about two product areas you already own.

  1. Ask how one major workflow works today
  2. Try one follow-up question for more detail
  3. Use /share to save a useful output
Day 3

Use it in a real product ritual

Bring the CLI into sprint planning, a PR review, or a release check so the habit anchors to a real workflow.

  1. Use /review before a team call
  2. Ask for risks in plain English
  3. Compare the answer with the engineering story
Day 5

Use planning mode for real scope

Before the next roadmap conversation, create one implementation plan so you can discuss dependencies before commitment.

  1. Switch to plan mode with Shift+Tab
  2. Describe a real feature ask
  3. Review the plan with engineering

Best next prompt to try

“I’m a PM. Explain the current onboarding flow in this codebase, what files own it, and what the riskiest part would be to change.”
